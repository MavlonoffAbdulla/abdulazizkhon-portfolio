# WORKFLOW.md — Рабочий процесс

> Как агент (Antigravity / Claude Code) должен вести разработку этого проекта:
> порядок действий, git-конвенции, порядок проверки перед сдачей этапа.

## Общий цикл работы над каждым этапом из PLAN.md
1. Прочитать текущий незакрытый пункт в `PLAN.md`.
2. Свериться с `SKILLS.md` (конвенции кода) и `DESIGN.md` (визуальные токены).
3. Реализовать минимально достаточный код для пункта — не забегать вперёд по плану.
4. Проверить визуально (dev-сервер) на 3 ширинах экрана: 375px, 768px, 1440px.
5. Отметить пункт как выполненный `[x]` в `PLAN.md`.
6. Закоммитить с понятным сообщением (см. конвенцию ниже).

## Git-конвенция коммитов
Формат: `<тип>: <короткое описание на русском>`

Типы:
- `feat` — новая функциональность/секция
- `style` — правки вёрстки/стилей без изменения логики
- `fix` — исправление бага
- `i18n` — добавление/правка переводов
- `content` — правки текстов/данных кейсов
- `chore` — конфигурация, зависимости, инфраструктура

Примеры:
```
feat: добавлена секция Hero с CTA-кнопками
i18n: переведена секция Услуги на UZ и EN
content: добавлен кейс warehouse-системы в portfolio
```

## Порядок при добавлении нового кейса в портфолио
1. Добавить объект в `src/data/projects.js` с непереводимыми свойствами (поля: `id`, `category`, `stack`, `image`).
2. Добавить соответствующие текстовые ключи в `src/locales/ru.json`, `uz.json`, `en.json` по пути `portfolio.projects.<id>.*` (поля: `title`, `description`, `task`, `solution`, `result`). Структура ключей должна быть идентична во всех трёх файлах.
3. Проверить, что карточка появляется в нужном фильтре категории.
4. Проверить, что модальное окно/детальный вид кейса открывается корректно с корректным фокусом и доступностью.

## Порядок при добавлении нового языка в будущем (если понадобится)
1. Создать `src/locales/<код_языка>.json` с той же вложенной структурой ключей, что и в `ru.json`.
2. Добавить язык в конфиг `i18next` (`src/i18n.js`).
3. Добавить пункт в `LanguageSwitcher.jsx`.
